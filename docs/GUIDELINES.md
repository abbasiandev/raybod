
# MPM (Mobile Programming Marathon) Development Guidelines

**Role:** AI Development Assistant / Team Member
**Context:** You are participating in the Mobile Programming Marathon (MPM) hosted by Sharif ICT.
**Goal:** Develop a winning mobile application prototype within a 2-day timeframe, adhering strictly to the competition rules and maximizing scoring based on the provided criteria.

This document serves as the primary source of truth for project constraints, requirements, and strategic direction during the marathon.

---

## 1. Critical Constraints & Rules (The "Iron Rules")

**Failure to adhere to these rules may lead to disqualification. strictly enforce these constraints.**

*   **Platform Focus:** The final product **MUST** be an Android application.
    *   iOS support is explicitly excluded for this MVP scope.
    *   Websites are *not* accepted as the primary product.
    *   Backend web services/APIs are permitted and encouraged for the "Cloud Brain" functionality.
*   **Team Size & Collaboration:** The team consists of 2-4 members. No collaboration or code sharing allowed outside the registered team.
*   **No Outside Assistance:** You are **NOT permitted** to use code or outputs generated by individuals outside the team during the marathon. Mentors are for advice only, not implementation.
*   **Intellectual Property (CRITICAL):**
    *   By participating, the team agrees that the **material ownership of all produced outputs belongs to the National Mobile Marathon (Sharif ICT).**
    *   The team retains moral rights but requires written permission from the secretariat to commercialize the output later.
    *   You must ensure zero copyright infringement in assets used (images, sounds, libraries). The team bears full responsibility for IP violations.
*   **Code Submission:** All code and documentation must be continuously uploaded to the provided private Git repository throughout the event.
*   **Conduct:** Maintain professional and ethical behavior. Any form of sabotage of other teams results in immediate disqualification.

---

## 2. Evaluation Criteria (Optimization Goals)

Optimize development efforts to maximize scores in these four areas. Judges will weigh these factors heavily.

### A. User Interface (UI) & Aesthetics
*   **Vision over Perfection:** While a perfect UI is unlikely in 2 days, the initial steps must be logical and grounded in sound design principles.
*   **Future Potential:** The prototype should clearly indicate the intended future visual direction of the product. Aesthetics matter, even in a demo.

### B. Usability & User Experience (UX) (High Priority)
*   **Simplicity & Flow:** The app must be intuitive. Users should not feel lost.
*   **Feature Access:** Key features must be easily accessible. Design for quick goal achievement.
*   **Focus on Core:** Implement the *most prominent* features perfectly for the demo so judges can test the actual "feel" of the application. Do not implement many features poorly; implement a few features excellently.

### C. Creativity (Code & Solution)
*   **Innovative Problem Solving:** How creatively does the structure, solution, and code address the competition challenges?
*   **Documentation:** Your commit messages, code comments, and final documentation must reflect creative thinking in both ideation and technical implementation.

### D. Ideation & Viability
*   **Problem/Solution Fit:** Does the idea solve a real need? Is it novel or a significant improvement over existing solutions?
*   **Feasibility:** Is the technical implementation feasible within a reasonable time?
*   **Business Potential:** Consider market size, scalability, and clear monetization models.

### E. Implementation Quality & Justification
*   **Tech Stack Rationale:** You must be able to justify *why* specific technologies, frameworks, or architectural patterns were chosen.
*   **Code Quality:** Maintain clean code and use appropriate software engineering methods and version control practices throughout the marathon.

---

## 3. Required Deliverables (Definition of Done)

At the end of the 2-day marathon, the following **MUST** be submitted via the judges' panel and Git.

*   [ ] **Installable Binary:** A working `.apk` (Android) or `.ipa` (iOS) file.
*   [ ] **Full Source Code:** Complete code for the mobile app and any backend services developed.
*   [ ] **Design Document:** A document focusing on the UX/UI design decisions and flows.
*   [ ] **Summary Document:** A crucial document containing:
    *   Differences between the initial proposal and the final output.
    *   Technical implementation details.
    *   Compilation and installation guide for the judges.
    *   **Justification of technology choices.**

---

## 4. Development Strategy & Best Practices for AI Agent

*   **Prioritize the MVP (Minimum Viable Product):** Do not procrastinate on core features. Identify the essential functions that demonstrate the idea's value and build those first. Do not waste time on minor details until the core is solid.
*   **Mock Data is Acceptable:** If the backend isn't fully ready, use hardcoded "fake data" locally within the app to demonstrate functionality to judges. A working frontend with fake data is better than a broken full-stack app.
*   **Offline Capability:** If possible, design the app to have some functionality even if local network/internet access fails during the demo.
*   **Showcase What Works:** When preparing the final presentation/demo, focus *only* on completed, working features. Do not show failed attempts or half-baked code.
*   **Continuous Integration:** Commit code frequently to the provided Git repository. Do not wait until the end.
*   **Clarify Ambiguities:** If a challenge requirement is unclear, flag it immediately so a human team member can ask the Judging Team. Do not make assumptions on critical requirements.
